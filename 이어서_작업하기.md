# 다른 PC에서 이어서 작업하기 — 핸드오프 가이드

이 문서 하나로 **다른 컴퓨터에서 개발을 그대로 이어갈** 수 있습니다.
(작성 기준일: 2026-06-24)

## 0. 30초 요약
- **무엇**: ISP(통신사)별 유선 브로드밴드 품질을 공개 실데이터로 보여 주는 대시보드.
- **저장소**: `https://github.com/ai-dh-kim/ispq` (public)
- **배포 주소**: **https://ai-dh-kim.github.io/ispq/** (GitHub Pages 자동 배포)
- **구조**: GitHub Actions 수집기가 공개 API에서 데이터를 모아 `public/quality_data.json`으로 커밋 →
  프론트(React)는 그 raw URL을 fetch해서 그림. **API 토큰은 Actions 시크릿에만 있고 클라이언트엔 0.**
- **데이터는 전부 실데이터** (시뮬레이션 폴백은 토큰/캐시 없을 때만).

## 1. 새 PC에서 시작하기
```bash
# 1) 클론
git clone https://github.com/ai-dh-kim/ispq.git
cd ispq

# 2) 의존성
npm install            # Node 20+ 필요, 권장 24 (스크립트가 .ts 직접 실행: 타입 스트리핑)

# 3) 개발 서버 (데이터는 raw GitHub URL에서 자동 fetch → 자격증명 없이도 화면 확인 가능)
npm run dev            # http://localhost:5173 (자동 오픈)
```
기타 명령:
- `npm run build` — 타입체크 + 프로덕션 빌드(`dist/`)
- `npm run typecheck` — `tsc --noEmit`
- `npm run mock` — `public/quality_data.json` 로컬 재생성 (⚠️ §5 주의 — 자격증명 없으면 일부 시뮬로 덮어씀)

> **로컬 dev는 자격증명이 필요 없습니다.** 데이터를 raw URL에서 가져오기 때문. 자격증명은 데이터를 *생성*할 때만 필요하고, 그건 평소 GitHub Actions가 대신합니다(§4).

## 2. 기술 스택
- **React 18 + Vite 5 + TypeScript** (import는 명시적 `.ts/.tsx` 확장자)
- **ApexCharts**(react-apexcharts) 차트 · **html2canvas** 스크린샷(동적 import)
- 데이터 수집 스크립트는 `.ts`를 **Node 24 타입 스트리핑**으로 직접 실행(빌드 불필요)
- `vite.config.ts`: `base: './'` (Pages 서브경로 `/ispq/`에서 자산 상대경로 로드)

## 3. 데이터 파이프라인 (핵심 아키텍처)
```
[공개 API]  →  [GitHub Actions 수집기]  →  public/*.json (저장소에 커밋)  →  [프론트가 raw URL fetch]
Cloudflare      collect.yml (10분)          quality_data.json                 https://raw.githubusercontent.com/
M-Lab BigQuery  collect-mlab.yml (일1회)     mlab_cache.json                    ai-dh-kim/ispq/main/public/quality_data.json
Netflix         collect-netflix.yml (일1회)  netflix_cache.json
```
- 프론트는 **단 하나의 파일** `quality_data.json`만 읽습니다(라이브 API 직접호출 없음 → 토큰 노출 0, rate-limit 회피).
- **2단계 캐시**: 비싸거나 느린 소스(M-Lab BigQuery, Netflix)는 하루 1회 따로 수집해 `*_cache.json`에 저장 →
  10분짜리 `generate-mock.ts`가 그 캐시 + Cloudflare 실시간 호출을 합쳐 `quality_data.json` 조립.

### 데이터 포맷 (`src/types.ts`)
다중 해상도(티어) 컬럼형:
- 티어: `fine`(10분/2일) · `mid`(1시간/30일) · `coarse`(1일/365일)
- `series[ispId][metricId][tier] = [ v[], n[], k[] ]` (값 / 총샘플수 / 유효샘플수). 빈 버킷 `null`.
- 실측 percentile 데이터(Cloudflare/Netflix)는 표본수 미제공 → `n/k=null` (툴팁에 "실측값" 표기).
- `liveMetrics`: 실데이터가 연결된 지표 id 배열. `mode`: 'live'|'sim'.

## 4. GitHub Actions 워크플로 & 시크릿
| 워크플로 | 파일 | 주기 | 역할 |
|---|---|---|---|
| Refresh quality_data.json | `collect.yml` | 10분 + 수동 | Cloudflare 실호출 + 캐시 합쳐 `quality_data.json` 재생성·커밋 |
| Collect M-Lab daily cache | `collect-mlab.yml` | 일 1회 + 수동 | M-Lab BigQuery → `mlab_cache.json` |
| Collect Netflix daily cache | `collect-netflix.yml` | 일 1회 + 수동 | Netflix ISP Speed Index → `netflix_cache.json` |
| Deploy to GitHub Pages | `deploy.yml` | main push + 수동 | 코드 push 시 Vite 빌드 → Pages 배포 |

**저장소 시크릿** (Settings → Secrets and variables → Actions). 이미 설정돼 있음:
- `CLOUDFLARE_API_TOKEN` — Cloudflare Radar 읽기 토큰
- `MLAB_BQ_KEY` — GCP 서비스계정 키 **JSON 전체**(한 줄). M-Lab BigQuery 조회용
- `MLAB_BQ_PROJECT` — GCP 프로젝트 id (쿼리 과금 주체)

> ⚠️ **GCP 서비스계정 키 파일은 저장소·로컬 어디에도 커밋 금지.** `.gitignore`가 `*gserviceaccount*.json`·`mlab-bq/`·`.env` 차단. 키 원본은 원래 PC의 저장소 **바깥** 디렉터리에 있음 — 새 PC에서 로컬로 M-Lab을 돌리려면 그 키가 별도로 필요. **평소엔 Actions가 시크릿으로 돌리므로 로컬 키 불필요.**

## 5. ⚠️ 새 PC에서 `npm run mock` 주의
- 자격증명(env)이 없으면 Cloudflare는 **시뮬로 폴백**합니다. 그 상태로 만든 `quality_data.json`을 **커밋하면 실데이터를 시뮬로 덮어씁니다.**
- 따라서 로컬에서 화면만 볼 땐 굳이 `npm run mock` 하지 말고 `npm run dev`로 raw 데이터를 그대로 보세요.
- 데이터를 새로 만들 일이 있으면 **GitHub Actions에서 워크플로를 수동 실행**(Run workflow)하는 게 정석입니다.
  - 순서: `Collect M-Lab` / `Collect Netflix`(캐시 갱신) → `Refresh quality_data.json`.

## 7. 현재 지표 (13종, 전부 실데이터 연동)
| 출처 | 지표 id | 이름 | 단위 | cite | 비고 |
|---|---|---|---|---|---|
| Cloudflare | `latency` | 지연시간(RTT) | ms | A | IQI p50 |
| Cloudflare | `dnsResponse` | DNS 응답시간 | ms | A | IQI metric=DNS p50 |
| Cloudflare | `bandwidth` | 대역폭(기준) | Mbps | A | IQI 다운로드 중앙값. **체감속도≠회선용량**(ⓘ 주석) |
| Cloudflare | `p25Throughput` | 보장 처리량(하위25%) | Mbps | A | IQI p25. 체감속도 주석 |
| Cloudflare | `ipv6` | IPv6 채택률 | % | A | HTTP ip_version. **KR 유선 0% 근처는 실측**(ⓘ 주석) |
| M-Lab | `meanThroughput` | 평균 처리량 | Mbps | A | ⓘ 측정 한계 주석(단일 TCP·자기선택) |
| M-Lab | `peakCapacity` | 피크 처리량(상위10%) | Mbps | A | 분위수 꼬리 평균. "관측 피크"(ⓘ 주석) |
| M-Lab | `minRtt` | 최소 RTT | ms | A | p50 |
| M-Lab | `latencyFloor` | 지연 하한(하위10%) | ms | A | 분위수 꼬리 평균. 백본 경로 프록시 |
| M-Lab | `lossRate` | 손실률 | % | B | LossRate×100 |
| M-Lab | `nfHd` | HD 스트리밍 가능률(≥5Mbps) | % | C | 처리량 ≥5Mbps 비율(COUNTIF). 출처=M-Lab(파생) |
| M-Lab | `nf4k` | 4K 스트리밍 가능률(≥15Mbps) | % | C | 처리량 ≥15Mbps 비율(COUNTIF). 출처=M-Lab(파생) |
| Netflix | `nfSpeedIndex` | ISP Speed Index | Mbps | A | 월별, 항상 최근 180일 coarse 고정 |

- 제목 옆 **근거등급 태그** 표시: 직접측정(A·녹색)/집계(B)/파생(C·주황). `cite.grade` 기반(MetricSection).

- `mlabBased` 플래그(meanThroughput·peakCapacity·minRtt·latencyFloor·lossRate·nfHd·nf4k): 차트에 **지연 공지** + **X축을 최신 M-Lab 데이터에서 정지**(공개 ~2일 지연).
- 대상 ISP: 한국 3사(LG U+ / KT / SK브로드밴드) + 해외 각국 대표 10사. `src/data/isps.ts`.
- **LG U+**: ASN 2개(AS3786·AS17858)를 박스 하나에서 개별 선택. 둘 다 선택 시 합산(통합 `lgu`)으로 표시.
  - M-Lab 수집기는 ASN→ISP를 **1:다**로 매핑해 통합 `lgu`가 두 ASN 합산값을 받음(`collect-mlab.ts`).
- Y축: 보이는 X구간의 데이터 최대값이 축의 ~80%가 되도록 `niceCeil(max/0.8)`. 모바일(터치)은 줌/팬 비활성.

## 7-1. 사실성 검토 결과 (임원 보고 기준 · 2026-06-24 점검)
프로덕션 `quality_data.json` 직접 검증 — **13종 모두 진짜 실데이터(mode=live), 값 범위 타당, 시뮬을 실데이터로 라벨한 곳 없음.**
- **검증된 실측값 예**(kt, 최근 30일): latency 8.7~11.6ms, dns 8.4~17.6ms, bandwidth 43~63Mbps, meanThroughput 86~102Mbps, lossRate 0.8~3.0%, nfHd 87~99%, nfSpeedIndex 3.0~3.4Mbps(월별).
- **ipv6 KT·SKB ≈0%는 버그 아님**: 해외 ISP는 40~73%로 정상(단위 일관). KR 유선 IPv6가 실제로 낮음(LG U+ 39%). ⓘ 주석으로 명시함.
- **오독 방지 캡션 추가**: bandwidth·p25(체감속도≠회선용량), ipv6(KR 낮음 실측). meanThroughput·peakCapacity·latencyFloor엔 이미 측정 한계 주석.
- **정직성 개선(적용 완료)**:
  1. ✅ `nfHd`/`nf4k` 출처를 netflix→**mlab로 정정**(Netflix 측정값 아님), 이름도 'HD/4K 스트리밍 가능률(≥5/≥15Mbps)'로 명확화. Netflix 탭엔 실제 공개값 `nfSpeedIndex`만.
  2. ✅ **근거등급 화면 표시**: 직접측정(A)/집계(B)/파생(C) 태그를 제목 옆에 노출 → 직접측정과 파생을 한눈에 구분.
- **남은 판단거리(미적용, 선택)**:
  3. Cloudflare 희소 구간은 **forward-fill**(직전값 유지, n=null) — 보간이며 그 시점 독립 측정 아님.
  4. 통합 LG U+의 peakCapacity/latencyFloor는 ASN별 꼬리평균의 n가중 평균(근사). 상대 비교엔 무방.

## 8. 폐기된 접근 (다시 시도하지 말 것)
- **RIPE Atlas** — 프로브가 희소·지역 교란이 커서 ISP 품질 비교에 부적합 → **완전 제거**.
- **uploadThroughput / cwnd / pacingRate / httpErrorRate** — 시뮬뿐이라 제거.
  - 업로드는 M-Lab 별도 뷰(`unified_uploads`)라 쿼리 추가 시 BigQuery 스캔비용 ~2배 → 비용 때문에 보류(원하면 재추가 가능).
- **YouTube(Google Video Quality Report)** — 서비스 종료(404).
- **fast.com 직접 수집** — 러너 1대 회선만 측정, ISP별 아님. 그걸 통신사별 집계한 게 Netflix Speed Index.

## 9. 비용 메모 (M-Lab)
- BigQuery 스캔 ~36.86GB/회, 일 1회 → 월 ~1.1TB. 무료 1TB/월을 살짝 초과 → **월 약 $0.5~0.7**.
- 줄이려면 `collect-mlab.ts`의 `TIERS` coarse 기간 90d→60d로 축소.
- `CAP_GB=50` 안전장치: dry-run 예상 스캔이 50GB 초과면 자동 중단(과금 사고 방지).
