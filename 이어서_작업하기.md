# 다른 PC에서 개발 이어서 하기 — 핸드오프 가이드

이 문서는 **개발 환경을 다른 컴퓨터로 옮겨 이어서 작업**하기 위한 안내입니다.
(시연용 실행은 `README.md` / `대시보드 실행.bat` 참고)

> **다음 단계 = 실데이터 연동(Netflix).** 바로 §10 "실데이터 연동 가이드"부터 보면 됩니다.

## 1. 사전 준비 (새 PC에 설치)
- **Node.js LTS 20 이상** (권장 **24** — 모크 생성기가 `.ts`를 직접 실행, 타입 스트리핑 사용)
  - https://nodejs.org → LTS 설치
- **Git** (이력 이어가려면) — https://git-scm.com
- 코드 에디터 (VS Code 권장)

## 2. 프로젝트 가져오기 (둘 중 하나)
- **(A) zip 핸드오프:** `핸드오프_개발용.zip`(node_modules 제외, .git 포함)을 풀기
- **(B) Git 원격:** 원격 저장소가 있으면 `git clone <url>`

## 3. 개발 환경 셋업
```bash
cd fixed_broadband_quality_dashboard   # 폴더명은 상황에 맞게
npm install            # 의존성 설치 (node_modules 재생성)
npm run mock           # public/quality_data.json 생성 (zip에 포함돼 있으면 생략 가능)
npm run dev            # http://localhost:5173
```
기타: `npm run build`(타입체크+빌드), `npm run typecheck`

## 4. 기술 스택
- **React 18 + Vite 5 + TypeScript** (모든 소스 `.ts/.tsx`, import는 명시적 확장자)
- **ApexCharts**(react-apexcharts) — 차트
- **html2canvas** — 화면 스크린샷(동적 import)
- tsconfig: `allowImportingTsExtensions` + `noEmit`, `moduleResolution: bundler`

## 5. 데이터 구조 (중요 — 실데이터도 이 포맷을 따라야 함)
- 프론트는 **단일 정적 파일** `public/quality_data.json` 만 fetch (라이브 API 직접호출 없음).
- **다중 해상도(티어)** 한 파일, **컬럼형(배열) 포맷**:
  - `fine` 10분/2일(실제 원샘플 FR-03 절단) · `mid` 1시간/30일 · `coarse` 1일/365일(대표 집계)
  - 범위 선택 → 티어 자동, 버킷(10분/1시간/1일)은 티어 기본 이상만
- **JSON 스키마**(타입 정의: `src/types.ts`):
  ```jsonc
  {
    "generatedAt": "ISO8601", "mode": "sim" | "live", "lang": "ko",
    "tiers": { "fine": {"baseMin":10,"t":[epochMs,...]},
               "mid":  {"baseMin":60,"t":[...]},
               "coarse":{"baseMin":1440,"t":[...]} },
    "isps":    ["lgu","kt","skb", ...],   // src/data/isps.ts 의 id
    "metrics": ["meanThroughput", ..., "nfHd","nf4k","nfSpeedIndex"],
    "series": {
      "<ispId>": {
        "<metricId>": {
          "fine":  [ [v...], [n...], [k...] ],   // TierBlock: 값, 총샘플, 잔존샘플
          "mid":   [ [v...], [n...], [k...] ],
          "coarse":[ [v...], [n...], [k...] ]
        }
      }
    }
  }
  ```
  - 각 티어 배열은 해당 `tiers.<tier>.t` 시간축과 **인덱스 정렬**. 빈 버킷은 `null`.
  - `v`=절단평균(또는 비율%), `n`=총 원샘플, `k`=FR-03 절단 후 잔존. 추적성 툴팁이 이 n/k를 씀.
- 생성기: `scripts/generate-mock.ts` (현재 **시뮬레이션** 데이터). 실 연동 시 `simulate*` 부분만 교체.

## 9. 설계 결정 / 폐기된 접근 (다른 PC에서 재시도 금지)
- **YouTube(Google Video Quality Report) 제외** — 해당 서비스는 **종료됨**.
  `https://www.google.com/get/videoqualityreport/` 가 **HTTP 404**. 스크래핑할 페이지가 없음.
- **fast.com 직접 수집 안 함** — fast.com은 "실행한 그 기기"의 회선만 재는 **개인 속도측정 도구**(M-Lab NDT와 동류). CI/서버에서 돌리면 ISP별이 아니라 그 러너의 속도 1개만 나옴.
  → 사용자별 측정을 **통신사별로 집계해 이미 공개한 게 Netflix ISP Speed Index**. 그걸 쓰면 됨.
- **Gemini가 제안한 Python+Playwright 스크래퍼 / 평면 `google_vqr` 배열 노드 → 폐기.**
  기존 다중티어 컬럼형 포맷(§5)·TS 생성기와 충돌. 별도 차트/로더가 또 필요해지므로 채택 안 함.

## 10. 실데이터 연동 가이드 (Netflix) — **다음 작업**
사용자 방침: **실측 데이터를 본인 GitHub 저장소의 `public/quality_data.json`(§5 포맷)에 넣고, 프론트는 그 raw URL을 fetch.** 즉 별도 백엔드 없이 "데이터 파일만" 갈아끼우는 방식.

### 10-1. 어떤 지표를 어디서
| 지표 | 실데이터 출처 | 산출 방법 | 적합 티어 |
|---|---|---|---|
| `nfSpeedIndex` | **Netflix ISP Speed Index** (월별·ISP별 프라임타임 평균 Mbps, 한국) — 살아있음 | 인덱스 값 그대로(월 1회) | `coarse`(1일) — 월값을 해당 월 일자에 채움 |
| `nfHd` (≥5Mbps %) | **M-Lab** ndt7 처리량 (BigQuery 공개셋, ASN별) | 버킷별 (처리량 ≥ 5Mbps 샘플수)/(총샘플) ×100 | fine/mid/coarse 전부 |
| `nf4k` (≥15Mbps %) | 동일(M-Lab) | 버킷별 (처리량 ≥ 15Mbps)/(총) ×100 | fine/mid/coarse 전부 |

- 임계값 근거: 1080p≈5Mbps, 4K≈15Mbps. `rating_grade`(등급)는 `nfHd`%에서 `NF_GRADES`로 파생(코드가 자동).
- Netflix ISP Speed Index는 **JS 렌더 페이지**(`ispspeedindex.netflix.net`)라 단순 HTML fetch 불가 → 헤드리스(Playwright) 또는 페이지가 호출하는 백엔드 JSON을 직접 호출해 수집.
- M-Lab BigQuery: GCP 프로젝트 + 서비스계정 필요(무료 쿼리 월 1TB). ASN 목록(`src/data/isps.ts`의 asns) + 최근 기간으로 필터.

### 10-2. 코드에 붙이는 지점
1. **프론트 데이터 소스 URL** — `src/lib/settings.ts`의 `DEFAULT_SETTINGS.dataUrl` 을
   `https://raw.githubusercontent.com/<계정>/<repo>/main/public/quality_data.json` 로 변경.
   (또는 실행 중 "⚙ API 설정" 모달에서 URL 입력 — localStorage 영속.)
2. **수집/생성** — 둘 중 택1:
   - (간단) 본인이 만든 실측 `quality_data.json`을 그대로 GitHub에 push.
   - (자동) `scripts/generate-mock.ts`의 `simulateSamples`/`synthAggregate`에서
     nfHd/nf4k는 M-Lab 처리량 샘플로, nfSpeedIndex는 Netflix 인덱스 값으로 대체하고,
     `.github/workflows/collect.yml` 크론으로 재생성(시크릿: NETFLIX/MLAB).
3. **(선택) API 설정 모달에 Netflix 토큰칸** — 현재 `settings.ts`의 `ProviderId`는
   `'cloudflare'|'mlab'|'ripe'`만 있어 Netflix 항목이 없음. dataUrl 방식이면 불필요하나,
   모달에 노출하려면 `ProviderId`/`DEFAULT_SETTINGS.providers`/`ApiSettings.tsx`에 netflix 추가.

### 10-3. 검증 체크리스트
- `npm run mock` 후 `public/quality_data.json`의 `metrics`에 `nfHd/nf4k/nfSpeedIndex` 포함 확인.
- `npm run dev` → 출처 "Netflix 스트리밍 품질" 선택 → 차트 3개, 피크카드 등급 표시.
- `nfSpeedIndex` 값이 한국 실측 범위(대략 3~4Mbps대)인지, `nfHd`% 등급 매핑이 맞는지 확인.
